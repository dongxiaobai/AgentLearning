# Token 与 Tokenizer 完全指南（02）

本文整合 Token 概念、Tokenizer 与 BPE 原理及工程影响，一册讲清。

------------------------------------------------------------------------

## 一、Token 是什么？（最直观理解）

**Token = 大语言模型能够处理的最小文本单位。**

- 它通常是**子词（subword）**，不是完整的字或单词。
- **Token ≠ 字，≠ 单词，≠ 句子**：由分词器（Tokenizer）按词表切出来的片段，对应词表里的一个**整数编号（token id）**。
- 模型输入的不是 "北京天气" 这四个字，而是例如 `[1234, 5678]` 这样的数字序列，再经 Embedding 变成向量。

------------------------------------------------------------------------

## 二、为什么不能直接用「字」或「单词」？

| 方式 | 问题 |
|------|------|
| **用单词** | 词表极大（几十万甚至百万）、新词无法处理（OOV） |
| **用字** | 序列极长、表达效率低、训练慢 |
| **子词（Subword）** | 折中：常见词一个 token，生僻词拆成几个已知子词，词表可控、能处理未登录词 |

所以引入 **子词切分（Subword Tokenization）**，如 BPE、SentencePiece（见下文）。

------------------------------------------------------------------------

## 三、Token 是如何生成的？从文本到模型输入

```
文本（字符串）
  "杭州副业做什么赚钱？"
      ↓
【Tokenizer（如 BPE / SentencePiece）】
      ↓
Token 序列
  例如：["杭州", "副", "业", "做", "什么", "赚钱", "？"]
      ↓
【查词表】每个 token 对应一个编号
      ↓
Token ID 序列（整数）
  例如：[10342, 8881, 4921, 2210, 776, 9122, 13]
      ↓
【Embedding 层】每个 token id → 一个向量
      ↓
模型只处理：向量矩阵（不处理字符串、不理解字符）
```

**工程认知**：模型从头到尾不处理字符串，只处理**向量矩阵**；token 是中间的「离散符号 ID」。

------------------------------------------------------------------------

## 四、词表（Vocabulary）

- **词表** = 该模型所有可能 token 的集合，大小常见几万～十几万（如 32k、50k、100k+）。
- 每个 token 对应一个 ID（0 ～ 词表大小-1）；不同模型词表不同，同一句话在不同模型下 **token 数与边界可能不同**。
- 「多少字」不能直接等于「多少 token」，必须用**该模型自带的 tokenizer** 算。

------------------------------------------------------------------------

## 五、同一句话能切出几个 Token？（具体例子）

| 文本 | 约 token 数 | 说明 |
|------|-------------|------|
| 你好 | 1～2 | 中文常 1 字 1 token 或 2 字 1 token |
| 北京今天天气怎么样 | 约 5～12 | 依词表可能「北京」「今天」「天气」等 |
| Hello world | 2 | 英文常 1 词 1 token |
| 机器学习 | 2～4 | 可能「机器」「学习」或更碎 |
| 100 字中文 | 约 50～150 | 约 1 字 0.5～1.5 token |

**中文**：通常一个汉字 ≈ 1 个 token，但常见词可能被合并成一个 token（如「人工智能」可能 1 个或「人工」+「智能」），取决于 tokenizer 训练语料。

------------------------------------------------------------------------

## 六、Tokenizer 与 BPE 原理

### 为什么需要 Tokenizer？

模型不能处理字符串，只能处理整数。Tokenizer 负责：**文本 → token → token id**。

### BPE（Byte Pair Encoding）核心思想

- 从**字符**开始，逐步合并**出现频率最高**的字符对。
- 示例：`l o w` 与 `l o w e r` 中，先合并 `lo`、`ow` 等高频对，得到子词级别的 token。
- **优势**：词表大小可控、能处理未登录词、泛化好。
- 不同模型用不同 tokenizer（如 GPT 系列用 BPE，LLaMA 用 SentencePiece），所以**同一句话在不同模型下 token 数可能不同**。

------------------------------------------------------------------------

## 七、Token 的三个核心工程影响

| 影响 | 说明 |
|------|------|
| **1️⃣ 成本** | API 计费 = 输入 token + 输出 token；token 越多，成本越高。 |
| **2️⃣ 上下文长度** | 模型有最大 context window（如 4k、128k）；超过就无法输入。 |
| **3️⃣ 推理速度** | Transformer 对序列的复杂度与 token 数相关（如 attention 的 O(n²)）；token 越多，计算量越大、越慢。 |

------------------------------------------------------------------------

## 八、Token 与 Embedding 的关系

每个 token id 会映射到一个向量：**token_id → embedding vector**。  
例如：10342 → [0.12, -0.77, 1.33, ...]  
模型真正处理的是这些向量，不是字符串。

（用于 RAG 的「句向量」是另一类 Embedding 模型，见 08。）

------------------------------------------------------------------------

## 九、如何知道一段文本有多少 Token？

用该模型自带的 tokenizer 计算，例如：

```python
# OpenAI
import tiktoken
enc = tiktoken.encoding_for_model("gpt-4")
len(enc.encode("北京今天天气怎么样？"))  # 得到 token 数

# Hugging Face
from transformers import AutoTokenizer
tok = AutoTokenizer.from_pretrained("bert-base-chinese")
len(tok.encode("北京今天天气怎么样？"))
```

粗略经验：英文约 1 词 ≈ 1～1.5 token；中文约 1 字 ≈ 0.5～1.5 token。精确值必须用对应 tokenizer。

## 十、常见工程误区

- ❌ 误以为「字数」等于 token 数  
- ❌ 忽略 token 计费  
- ❌ 忽略 prompt 拼接导致 token 爆炸  
- ❌ 忽略历史对话导致 context 超限  

------------------------------------------------------------------------

## 十一、Token 在 Agent / RAG 中的影响

- **RAG**：chunk 切分大小要按 **token 数**控制，否则易超上下文上限。  
- **Agent**：每一轮 Thought / Observation 都消耗 token；设计不合理会导致成本飙升、性能下降、上下文溢出。  

------------------------------------------------------------------------

## 十二、与其他文档的关系

- **05_Transformer**：token id 经 Embedding 后进入 Transformer，见其数据流。  
- **07_RAG / 08_Embedding**：RAG 里拼进 prompt 的长度、Embedding 模型输入长度，都受 token 限制。  

------------------------------------------------------------------------

## 十三、总结

- **Token** = LLM 世界的「最小计算单位」，对应词表里的一个 id，通常是子词。  
- 从文本到模型：**文本 → Tokenizer → token 序列 → token id → Embedding → 向量**。  
- 工程上必须关注：**token 数量、token 成本、context 上限、tokenizer 差异**。  
- 能精准理解并控制 token，才算真正理解 LLM 工程。
