# Embedding 详解

Embedding（嵌入）指把**文本映射成固定长度的向量**，使「语义相近的文本在向量空间里距离更近」，从而支持相似度检索、推荐、聚类等。在 RAG 里，文档和问题都要先变成向量，才能做语义检索。

------------------------------------------------------------------------

## 一、Embedding 是什么？

**一句话**：把一段文字（词、句、段）映射成一个**固定维度的实数向量**。

例如：

- "猫" → [0.21, -0.88, 1.02, 0.33, ...]（如 768 维）
- "狗" → [0.19, -0.82, 0.98, 0.31, ...]（和「猫」较近）
- "今天天气" → [0.01, 0.45, -0.22, ...]（和「猫」较远）

**性质**：语义越相似，向量距离越近（或夹角越小）；这样就能用「向量相似度」代替「字面匹配」做检索。

------------------------------------------------------------------------

## 二、两种「Embedding」语境（别搞混）

| 语境 | 是什么 | 典型用途 |
|------|--------|----------|
| **LLM 内部的 Embedding 层** | Transformer 里把 token id 变成向量的那一层，输入是 token，输出是每个 token 的向量，维度 = hidden_dim | 模型前向计算、下一个 token 预测（见 05_Transformer） |
| **独立的 Embedding 模型 / API** | 专门「文本 → 一个向量」的模型或接口，输入是一段文本，输出是一个句向量（或段向量） | RAG 检索、语义搜索、聚类、推荐（见 07_RAG） |

本文主要讲**第二种**：用于 RAG / 语义检索的 **Embedding 模型**。LLM 内部的那一层在 05 里已经写过。

------------------------------------------------------------------------

## 三、Embedding 模型在做什么？

- **输入**：一段文本（一句话、一个段落、一个 chunk）。
- **输出**：一个固定长度的向量，如 768 维、1024 维、1536 维（依模型而定）。
- **过程**：文本先 tokenize，再经过一个**编码器**（多为 Transformer 的 encoder 或小模型），对整段做编码，最后通过 **pooling**（如取 [CLS]、或对所有 token 向量做 mean）得到**一个**向量。

所以：Embedding 模型 = **文本编码器 + 池化 → 一个向量**；不做「生成下一个词」，只做「表示这段文本」。

------------------------------------------------------------------------

## 四、维度与归一化

| 概念 | 说明 |
|------|------|
| **维度 (dimension)** | 向量长度，如 768、1536。维度高一般表达力更强，但占存储、算相似度更贵。不同模型维度不同，不能混用。 |
| **归一化 (normalization)** | 很多模型输出会做 L2 归一化，使向量长度为 1。此时「余弦相似度」= 「内积」(cos = 点积 / (\|a\|\|b\|)，若 \|a\|=\|b\|=1 则 cos = 点积)。检索时用内积或余弦等价，且内积更快。 |

------------------------------------------------------------------------

## 五、相似度怎么算？

检索时用「查询向量」和「文档向量」做相似度，常用：

| 方式 | 公式/说明 | 适用 |
|------|------------|------|
| **余弦相似度** | cos(θ) = (A·B) / (\|A\| \|B\|)，范围 [-1, 1] | 向量未归一化时常用 |
| **内积 (点积)** | A·B；若已 L2 归一化，则内积 = 余弦相似度 | 向量库常用，计算快 |
| **欧氏距离** | \|A - B\|；距离越小越相似 | 有的向量库用距离而非相似度，注意取负或取反 |

**要点**：文档和问题必须用**同一套 Embedding 模型**，否则向量不在同一空间，相似度无意义。

------------------------------------------------------------------------

## 六、在 RAG 里怎么用？（对应 07）

- **离线**：把知识库里的每个 chunk 用 Embedding 模型变成向量，写入向量数据库（每条记录 = 向量 + 原文或原文 ID）。
- **在线**：用户问题用**同一个** Embedding 模型变成向量，在向量库里做相似度检索（如 Top-K 内积最大），取回对应的 chunk 原文，再拼进 prompt 给 LLM。

所以：**同一模型、同一维度**，才能保证「问题」和「文档」在同一个向量空间里可比。

------------------------------------------------------------------------

## 七、常见 Embedding 模型 / API

| 类型 | 示例 | 特点 |
|------|------|------|
| **OpenAI** | text-embedding-3-small / ada-002 | API 调用，按 token 计费，多语言尚可，维度可选 |
| **开源句子向量** | sentence-transformers（如 all-MiniLM-L6-v2, paraphrase-multilingual） | 本地部署，免费，多语言有专门模型 |
| **中文/多语言** | BGE（BAAI）、M3E、GTE、E5 等 | 针对中文或多语言优化，常用来做 RAG |
| **厂商/云** | 各云厂商的 embedding 接口 | 与自家向量库、RAG 产品配套 |

选型时看：**语言**（中/英/多语言）、**维度与性能**、**是否本地部署**、**与现有向量库的兼容性**。

## 八、具体例子

### 例子 1：同一模型、不同文本的向量与相似度（示意）

用同一套 Embedding 模型（例如 768 维），对三句话编码后，相似度大致会呈现这样的关系（数值仅为示意）：

| 文本 A | 文本 B | 余弦相似度（约） | 说明 |
|--------|--------|------------------|------|
| 北京今天天气怎么样 | 北京今天天气如何 | 0.92～0.98 | 同义改写，非常接近 |
| 北京今天天气怎么样 | 上海明天会下雨吗 | 0.5～0.7 | 都是天气相关，有一定相似 |
| 北京今天天气怎么样 | 星巴克在中国有多少家店 | 0.1～0.3 | 语义不同，相似度低 |

向量本身是一长串数字，例如（仅展示前 5 维）：

- "北京今天天气怎么样" → [0.02, -0.31, 0.15, 0.44, -0.08, ...]（共 768 维）
- "北京今天天气如何"   → [0.03, -0.29, 0.14, 0.42, -0.07, ...]（共 768 维）

维度由模型决定（768、1536 等），不同模型不能混比。

### 例子 2：几个具体 Embedding 模型

| 模型名 | 维度 | 典型用法 | 备注 |
|--------|------|----------|------|
| **text-embedding-3-small** (OpenAI) | 1536（可缩） | 英文/多语言 RAG、检索 | API 调用，按 token 计费 |
| **text-embedding-ada-002** (OpenAI) | 1536 | 同上，旧版 | 仍常用 |
| **all-MiniLM-L6-v2** (sentence-transformers) | 384 | 英文句子相似、小规模 RAG | 本地、轻量 |
| **paraphrase-multilingual-MiniLM** | 384 | 多语言（含中文）句子相似 | 本地 |
| **BAAI/bge-small-zh** | 512 | 中文 RAG、检索 | 开源，适合中文 |
| **BAAI/bge-large-zh** | 1024 | 中文高精度 | 更大、更准、更慢 |

同一模型内：**同一段文本多次编码，得到的向量一致**（无随机性）；**不同模型**得到的维度和数值都不同，不能混用。

### 例子 3：调用方式示意（Python）

**OpenAI API：**

```python
# 一段文本 → 一个向量（1536 维）
from openai import OpenAI
client = OpenAI()
res = client.embeddings.create(model="text-embedding-3-small", input="北京今天天气怎么样？")
vec = res.data[0].embedding  # list of 1536 floats
```

**sentence-transformers（本地）：**

```python
from sentence_transformers import SentenceTransformer
model = SentenceTransformer("paraphrase-multilingual-MiniLM-L12-v2")
vec = model.encode("北京今天天气怎么样？")  # 例如 shape (384,)
# 多句可批量
vecs = model.encode(["问题1", "问题2", "文档1"])
```

**RAG 检索时**：用 `vec` 去向量库里做 Top-K 相似度搜索（内积或余弦），取回的就是「和这句话最像」的文档 chunk。

## 九、工程上要注意什么

| 点 | 说明 |
|----|------|
| **文档与查询用同一模型** | 否则向量空间不一致，检索失效 |
| **长度限制** | 多数模型有最大 token 数（如 512、8192），超长要截断或分段再 pooling |
| **Batch** | 离线建库时批量编码可显著加速；在线查询通常单条或小 batch |
| **维度与存储** | 维度 × 条数 × 4 字节（float32）≈ 向量库体积；高维更占空间、检索稍慢 |
| **归一化** | 若模型输出已归一化，向量库用内积即可；否则用余弦或库支持的度量 |

## 十一、小结

- **Embedding** = 文本 → 固定维度的向量；语义相近则向量距离近。
- **两种语境**：LLM 内部 token embedding vs 独立 Embedding 模型（RAG 用后者）。
- **RAG 中**：文档和问题用**同一模型**得到向量，在向量库里按相似度检索，再拼 prompt 给 LLM。
- **工程**：选模型看语言与长度、注意维度与归一化、建库用 batch、查询与文档同一套模型。
