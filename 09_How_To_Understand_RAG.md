# 如何理解 RAG

RAG（Retrieval Augmented Generation）= 检索增强生成。用一句话概括：**先查、再拼、再生成**——从外部知识里检索相关内容，拼进 prompt，让 LLM 基于这些内容生成回答。

------------------------------------------------------------------------

## 一、为什么需要 RAG？

LLM 的局限（对应你学的 LLM 本质）：

- 知识有截止时间，不知道最新或私有数据。
- 容易幻觉：概率生成，不验证事实。
- 上下文有限：不能把整座知识库都塞进 prompt。

RAG 的目标：

> 不重新训练模型，给模型「外挂」可更新的外部知识；需要时先检索，再让模型基于检索结果生成。

------------------------------------------------------------------------

## 二、RAG 在做什么？（一句话）

**把「相关文档片段」检索出来，拼进当前问题的 prompt，让模型「基于这些片段」而不是仅凭参数里的知识来回答。**

所以：RAG 不是让模型「学会」新知识，而是**每次回答前，临时把相关知识塞进上下文**。

------------------------------------------------------------------------

## 三、RAG 流程（从文档到回答）

```
[离线] 文档/知识库
    ↓
切分 (chunk)
    ↓
向量化 (Embedding)
    ↓
写入向量数据库

[在线] 用户问题
    ↓
问题向量化 (同一套 Embedding)
    ↓
在向量库中检索 Top-K 相似片段
    ↓
将检索结果拼进 Prompt（如 system 或 user 里）
    ↓
调用 LLM 生成
    ↓
返回回答
```

**要点**：检索用的是「语义相似度」（向量距离），不是关键词匹配；拼进 prompt 的是「原文片段」，模型只是读这些文本再生成。

------------------------------------------------------------------------

## 四、核心组件

| 组件 | 作用 |
|------|------|
| **文档 / 知识源** | 要检索的对象：内部文档、帮助页、数据库记录等 |
| **Chunk 切分** | 把长文档切成小块（如 500 字 + overlap），便于检索和放入上下文 |
| **Embedding 模型** | 把文本变成向量，同一模型用于「文档」和「问题」，才能算相似度 |
| **向量数据库** | 存文档向量，支持相似度检索（如 Milvus、Faiss、pgvector、厂商向量库） |
| **检索逻辑** | 用「问题向量」查 Top-K 最相似的 chunk，有时加过滤（时间、来源） |
| **Prompt 拼装** | 把检索到的片段 + 用户问题，按固定格式拼成发给 LLM 的 messages |
| **LLM** | 基于「检索结果 + 问题」生成回答，不改变模型参数 |

------------------------------------------------------------------------

## 五、关键理解

- **检索 ≠ 生成**：检索只负责「找出来」；生成只负责「根据找出的内容写答案」。两者解耦。
- **相似度用向量**：问题和文档都用同一套 Embedding 变成向量，在向量空间里找最近的，即「语义检索」。
- **Chunk 设计很重要**：块太大则噪声多、超长，太小则语义不完整；常带 overlap 避免断句割裂。
- **RAG 不改变模型**：不微调、不训练，只改「输入给模型的内容」（即 prompt 里多了一段检索结果）。

------------------------------------------------------------------------

## 六、和「纯 LLM」「Agent」的关系

| 对比 | 纯 LLM | RAG | Agent |
|------|--------|-----|-------|
| 知识来源 | 训练数据（参数内） | 检索到的文档（拼进 prompt） | 可包含 RAG + 工具调用等 |
| 典型流程 | 问 → 答 | 问 → 检索 → 拼 prompt → 答 | 问 → 可能先 RAG 再工具 → 多步 → 答 |

RAG 可以单独用（问 → 检索 → 生成），也可以作为 Agent 的一步：Agent 决定「需要查知识库时」就调 RAG，把检索结果当上下文再继续推理或调工具。

------------------------------------------------------------------------

## 七、工程上要注意什么（检索细化）

### 1. Embedding 与 LLM 的匹配

- 多语言、专业术语时，选合适的 Embedding 模型（如中文可选 BGE、m3e、GTE、E5 等），再在小规模标注集上比较检索效果。

### 2. Chunk 策略

- **大小**：常见 200～800 字 / 500～2000 token 级别，按上下文上限与文档结构调整。
- **Overlap**：如窗口 500 字、重叠 50～100 字，避免一句话被硬切断导致语义缺失。
- **边界**：尽量按段落/标题/函数等「自然边界」切块，避免无意义的截断。

### 3. 检索数量 K

- K 太小 → 可能漏掉关键片段；K 太大 → 噪音多、token 成本高。
- 一般从 **K = 3～8** 起步，结合少量标注问题观察「正确片段是否在 Top-K 中」来微调。

### 4. Hybrid 检索（BM25 + 向量）

- 有些问题包含很关键的关键词/编号（如「表 A1」「订单 1234」），单靠语义向量不一定稳。
- 做法：分别用 **关键词检索（BM25）** 和 **向量检索** 各取 Top-N，再合并打分或取并集后再按某个规则排序。
- 效果：兼顾「精确匹配」与「语义相近」，对结构化标号、短关键词问题更 robust。

### 5. 重排（Re-ranking）

- 第一步：向量检索取 Top-K（如 20）。
- 第二步：用更强的重排模型（如 cross-encoder 或 LLM）对这 K 条逐条算相关性分数，再选前 M 条（如 5）拼进 prompt。
- 原则：**“先粗后精”**——大规模检索用快的近似向量搜索，小规模候选上用贵但准的模型精排。

### 6. 检索质量与生成质量的区分

- **检索质量**：看「该用的片段」有没有被 Top-K 检出来，以及排序是否合理。
- **生成质量**：模型是否只根据上下文回答，有没有胡编。
- 调 RAG 时，先看检索是否把对的东西找回来，再去调 prompt/生成；否则容易错怪模型。

### 7. 防幻觉与格式约束

- 在 prompt 里明确写「**仅根据以下内容回答**，如内容不足请回答不知道」，并要求引用片段。
- 对需要结构化输出的场景，结合 Chat API 的 Function Calling / JSON schema 限制输出格式。

------------------------------------------------------------------------

## 八、小结

- **RAG = 检索 + 拼进 prompt + 生成**；理解 RAG 就是理解这三步以及各自用的模块（切分、Embedding、向量库、拼 prompt、LLM）。
- **检索细化**：通过合理的 chunk、K 值、Embedding 选择、hybrid 检索与重排，提升「召回率 + 精度」，减少噪音与成本。
- **本质**：给模型外挂可更新的外部记忆，用检索在「需要时」把相关片段塞进上下文，让模型基于事实生成。
- **和 Agent**：RAG 可以是 Agent 的一种能力（查知识库）；Agent 还可以再结合工具、多步规划等。
