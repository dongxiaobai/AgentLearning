# 一个具体问题：从输入到输出的完整步骤与模块

以用户问「北京今天天气怎么样？」且系统配置了「查天气」工具为例，从输入到最终回复，逐步列出**做了什么事**、**用到了哪些模块/工具/模型**。

------------------------------------------------------------------------

## 场景设定

- **用户输入**：北京今天天气怎么样？
- **能力**：Agent，带一个工具 `get_weather(city, date)`
- **模型**：假设走 OpenAI Chat API（其他厂商流程类似）

------------------------------------------------------------------------

## 第一步：接收用户输入

| 项目 | 说明 |
|------|------|
| **发生的事** | 用户在前端输入文字并点击发送 |
| **执行的模块** | 前端 / 客户端（浏览器、App、API 客户端） |
| **输出** | 字符串 `"北京今天天气怎么样？"` 被发往后端（例如 JSON body 里的 `messages`） |

------------------------------------------------------------------------

## 第二步：会话与 Prompt 编排（应用层）

| 项目 | 说明 |
|------|------|
| **发生的事** | 后端把「系统设定 + 历史对话（若有）+ 当前用户消息」拼成 Chat 用的消息列表；若定义了工具，则准备好工具列表 |
| **执行的模块** | 应用服务 / 编排层（你写的 Agent 代码、或 LangChain 等框架的 prompt + 对话管理） |
| **输入** | 用户消息、历史消息、系统 prompt 模板、工具定义（name, description, parameters） |
| **输出** | 例如：<br>• `messages`: [{role: "system", content: "你是助手，可调用 get_weather 查天气。"}, {role: "user", content: "北京今天天气怎么样？"}]<br>• `tools`: [{ type: "function", function: { name: "get_weather", description: "...", parameters: {...} } }] |

**涉及「工具」**：无；这里只是**工具声明（schema）**，供 API 传给模型，尚未执行真实查天气。

------------------------------------------------------------------------

## 第三步：调用 Chat API（LLM 接口层）

| 项目 | 说明 |
|------|------|
| **发生的事** | 应用层把上一步的 `messages` + `tools` 发给模型服务（如 OpenAI） |
| **执行的模块** | HTTP 客户端 / SDK（如 `openai.ChatCompletion.create()`）、模型服务网关 |
| **输出** | 请求到达模型服务；若走 OpenAI，内部会把 `messages` 转成带特殊 token 的 prompt 文本（见 04_Chat_API） |

------------------------------------------------------------------------

## 第四步：服务端拼成最终 Prompt 文本

| 项目 | 说明 |
|------|------|
| **发生的事** | 模型服务把 role + content 转成该模型约定的一条长字符串（含 `<|system|>`, `<|user|>`, `<|assistant|>` 等） |
| **执行的模块** | 模型服务内部的 **Prompt 格式化 / 模板模块**（各厂商实现不同） |
| **输出** | 一条完整 prompt 字符串，例如：<br>`<|system|>你是助手，可调用 get_weather 查天气。<|user|>北京今天天气怎么样？<|assistant|>`（实际格式依模型而定） |

------------------------------------------------------------------------

## 第五步：Tokenization（分词）

| 项目 | 说明 |
|------|------|
| **发生的事** | 把上一步的 prompt 字符串切成 token，再映射成 token id 序列 |
| **执行的模块** | **Tokenizer**（如 Hugging Face `transformers` 的 tokenizer、或 OpenAI 的 tiktoken） |
| **输入** | 完整 prompt 字符串 |
| **输出** | 一维整数数组，如 `[101, 2345, 6789, ...]`，长度 = 序列长度 L（对应 02_Tokenizer_BPE） |

------------------------------------------------------------------------

## 第六步：模型前向计算（Transformer + LM Head）

| 项目 | 说明 |
|------|------|
| **发生的事** | token id → Embedding → 位置编码 → N 层 Transformer（Attention + FFN）→ 取最后位置的表示 → LM Head → 得到词表上的概率分布（对应 01_Self_Attention、05_Transformer） |
| **执行的模块** | **Embedding 层**、**Transformer 各层**、**LM Head**；由 **推理引擎**（如 PyTorch、vLLM、厂商自研引擎）在 GPU/CPU 上执行 |
| **输入** | token id 序列 `[B, L]`（这里 B=1 或小 batch） |
| **输出** | 下一个 token 的概率分布 P(next_token \| context)；若启用「工具调用」，则模型可能输出特殊结构（如 `tool_calls`），而不是普通文本 |

**说明**：Function Calling 时，模型内部仍是「预测下一个 token」；服务端会把连续生成的 token 解析成「是否包含 tool_calls」的结构化结果，再通过 API 返回。

------------------------------------------------------------------------

## 第七步：解码与结构化解析（模型服务内）

| 项目 | 说明 |
|------|------|
| **发生的事** | 根据概率分布做采样/贪婪得到下一个 token，自回归生成直到遇到停止条件；若为 tool_calls 模式，则解析生成内容为 `tool_calls`（name, arguments 等） |
| **执行的模块** | **解码器 / 采样模块**（temperature、top_p 等）、**输出解析模块**（把模型输出转成 API 约定的 JSON，如 `tool_calls`） |
| **输出** | API 响应，例如：<br>• `message.role = "assistant"`<br>• `message.content = null` 或一段思考文字<br>• `message.tool_calls = [{ id: "xxx", type: "function", function: { name: "get_weather", arguments: "{\"city\":\"北京\",\"date\":\"今天\"}" } }]` |

------------------------------------------------------------------------

## 第八步：应用层收到响应并解析

| 项目 | 说明 |
|------|------|
| **发生的事** | 你的 Agent 代码收到 API 返回的 `message`；若存在 `tool_calls`，则决定要执行哪些工具、用什么参数 |
| **执行的模块** | 应用服务 / Agent 编排层（你写的逻辑或框架的 Agent 执行器） |
| **输出** | 解析出：需要调用 `get_weather(city="北京", date="今天")` |

------------------------------------------------------------------------

## 第九步：执行工具（真正调用的「工具」）

| 项目 | 说明 |
|------|------|
| **发生的事** | 根据 name 找到注册的 `get_weather` 实现，传入解析出的参数并执行（例如调天气 API 或查库） |
| **执行的模块** | **你实现的工具函数** 或 **第三方 API 客户端**（如天气 API）、数据库驱动等 |
| **输入** | `city="北京", date="今天"` |
| **输出** | 工具返回结果，如字符串：`"北京今天晴，15～22℃"` |

------------------------------------------------------------------------

## 第十步：把工具结果拼回对话并再次请求模型

| 项目 | 说明 |
|------|------|
| **发生的事** | 把「assistant 的 tool_calls」和「tool 的返回」按 API 约定追加到 `messages`，再次调用 Chat API |
| **执行的模块** | 应用层 / Agent 编排层（拼消息）、HTTP 客户端 / SDK（再次请求） |
| **新的 messages 示例** | [ system, user, assistant(tool_calls), tool(get_weather 的返回), assistant(空，等模型补全) ] |

此后：**重复第四步～第七步**（服务端拼 prompt → tokenize → 模型前向 → 解码与解析）。  
这一次模型看到的上下文中已经包含「北京今天晴，15～22℃」，通常不再发起 tool_calls，而是直接生成一段回复文本。

------------------------------------------------------------------------

## 第十一步：模型第二次返回（最终文本回复）

| 项目 | 说明 |
|------|------|
| **发生的事** | 模型输出纯文本回复，无 `tool_calls`；API 返回 `message.content = "北京今天晴，15～22℃，适合出行。"` 之类 |
| **执行的模块** | 同第六步、第七步（Tokenizer、Transformer、解码器、输出解析） |

------------------------------------------------------------------------

## 第十二步：返回给用户

| 项目 | 说明 |
|------|------|
| **发生的事** | 应用层把最终 `message.content` 返回给前端；前端展示给用户 |
| **执行的模块** | 应用服务（构造 HTTP 响应）、前端 / 客户端（渲染 UI） |

------------------------------------------------------------------------

## 整体流程简图（按模块归类）

```
[用户] 输入「北京今天天气怎么样？」
    ↓
[前端] 发送到后端
    ↓
[应用/编排] 拼 messages + tools 定义
    ↓
[HTTP/SDK] 调用 Chat API
    ↓
[模型服务] Prompt 格式化 → [Tokenizer] → [Transformer + LM Head] → [解码/解析]
    ↓
返回：tool_calls = get_weather(北京, 今天)
    ↓
[应用/编排] 解析 tool_calls
    ↓
[工具实现] 执行 get_weather → 得到「北京今天晴，15～22℃」
    ↓
[应用/编排] 把结果拼进 messages，再次调用 Chat API
    ↓
[模型服务] 再次 Tokenizer → Transformer → 解码 → 返回纯文本
    ↓
[应用] 把最终文本返回 [前端] → [用户] 看到回复
```

------------------------------------------------------------------------

## 模块/工具/模型对照表

| 步骤 | 做的事 | 用到的模块/工具/模型 |
|------|--------|----------------------|
| 1 | 接收输入 | 前端、客户端 |
| 2 | 拼 prompt、带工具定义 | 应用层、编排层（或 LangChain 等） |
| 3 | 发请求 | HTTP 客户端、SDK、模型网关 |
| 4 | 转成 prompt 文本 | 模型服务的 Prompt 格式化 |
| 5 | 文本 → token id | **Tokenizer**（tiktoken / transformers） |
| 6 | 算下一个 token 概率 | **Embedding、Transformer、LM Head**（推理引擎） |
| 7 | 采样 + 解析 tool_calls | 解码器、输出解析 |
| 8 | 决定执行哪个工具 | 应用层、Agent 逻辑 |
| 9 | 真正查天气 | **工具**：get_weather 实现、天气 API 等 |
| 10 | 拼结果、再请求 | 应用层、HTTP/SDK |
| 11 | 第二次模型推理 | 同上：Tokenizer、Transformer、解码 |
| 12 | 把回复给用户 | 应用层、前端 |

------------------------------------------------------------------------

## 小结

- **只和「文本→概率」有关的**：Tokenizer、Embedding、Transformer、LM Head、解码器（对应 01、02、05）。
- **和「怎么问模型、怎么用结果」有关的**：Prompt 格式化、应用编排、工具定义与执行、多次 API 调用（对应 04、Agent 开发）。
- **真正被调用的「工具」**：只有第九步的 `get_weather` 这类你实现的函数或外部 API；**模型本身不是工具**，而是根据 prompt 和 tool 定义「决定要不要用工具、用什么参数」，由应用层去真正执行。

这样针对「一个具体问题」，从输入到输出每一步、每个模块/工具/模型就都对应上了。
